# Git 提交信息规范

本项目采用固定英文标签 + 中文摘要 + 中文编号说明的提交格式，优先参考 Conventional Commits，并结合当前工程习惯做简化约定。

## 推荐格式

```text
tag: 中文概要

1. 修改点一
2. 修改点二
3. 修改点三
```

## 书写规则

1. `tag` 使用固定值，不新增自定义标签。
2. 标题使用 `tag: 中文概要`，不建议使用 `[tag]` 形式；如需兼容旧风格，应仅用于历史提交，不作为新提交规范。
3. `中文概要` 用一句话说明本次提交的核心目的，避免使用“修改代码”“更新内容”这类空泛描述。
4. 正文使用 `1.` `2.` `3.` 按条列出主要修改，优先按“功能 / 测试 / 构建或文档”组织。
5. 一个提交只表达一个主题；若同时包含功能、重构、构建等多个独立主题，应拆分为多个提交。

## 固定标签

- `feat`：新增功能或新增模块能力
- `fix`：修复缺陷或纠正错误行为
- `refactor`：重构实现，但不改变对外行为
- `test`：新增或调整测试，不引入业务功能变更
- `docs`：文档更新
- `build`：构建系统、依赖、CMake、编译选项、打包相关修改
- `chore`：杂项维护，例如目录整理、脚本清理、非功能性配置调整
- `ci`：CI/CD 流水线相关修改
- `perf`：性能优化
- `revert`：回滚已有提交

## 标签选择建议

- 改动重点是“新增能力”时使用 `feat`
- 改动重点是“修复错误”时使用 `fix`
- 仅调整 `CMakeLists.txt`、`vcpkg.json`、编译参数或依赖时使用 `build`
- 仅补充或调整测试用例时使用 `test`
- 仅整理代码结构且不改变行为时使用 `refactor`
- 仅清理模板残留、无明确功能归属时使用 `chore`

## 快速复制模板

可先复制下列模板，再替换标题摘要与正文内容；若实际改动少于 3 条，可按需删减。

### `feat`

```text
feat: 新增某项能力

1. 新增/调整……
2. 补充/完善……
3. 处理……相关配置或兼容性问题。
```

### `fix`

```text
fix: 修复某个问题

1. 修正……在……场景下的错误行为。
2. 调整……逻辑，避免……问题再次出现。
3. 补充相关验证或回归测试。
```

### `build`

```text
build: 调整构建与依赖配置

1. 更新……构建配置或依赖声明。
2. 统一……目标、链接关系或编译参数。
3. 修正……环境下的构建一致性问题。
```

### `test`

```text
test: 补充某模块测试覆盖

1. 增加……正常场景测试。
2. 增加……边界或异常场景测试。
3. 完善……回归验证，提升可验证性。
```

### 摘要推荐写法

- 动词 + 对象：如“补充 utility 边界测试”“修复 numeric_cast 越界判断”
- 动词 + 对象 + 目的：如“统一模块构建配置以简化接入”
- 避免空泛摘要：如“修改代码”“更新内容”“处理问题”

## 推荐示例

### 示例 1：初始化工程并新增基础模块

```text
feat: 初始化 xin 工程并新增 utility 模块

1. 将项目从模板工程重命名为 `xin`，更新 `README.md`、顶层 `CMakeLists.txt` 和基础编译配置。
2. 新增 `src/common.utility.cppm` 模块，提供 `numeric_cast`、`string_cast`、`to_uppercase`、`to_lowercase` 及模板特征工具等基础能力。
3. 新增 `src/common.utility.test.cpp` 测试用例，并在 `src/CMakeLists.txt`、`vcpkg.json` 中补充测试目标与相关依赖，同时清理模板残留文件。
```

### 示例 2：仅调整构建配置

```text
build: 统一第三方依赖与模块构建配置

1. 调整 `CMakeLists.txt` 中第三方依赖查找方式，统一工程构建入口。
2. 更新 `src/CMakeLists.txt` 的模块目标链接配置，确保测试目标与库目标依赖关系清晰。
3. 同步 `vcpkg.json` 依赖声明，避免构建环境与源码配置不一致。
```

### 示例 3：仅补充测试

```text
test: 补充 utility 模块边界场景测试

1. 增加 `numeric_cast` 对空字符串、越界输入和部分匹配输入的测试覆盖。
2. 增加大小写转换函数在空字符串和特殊字符场景下的验证。
3. 补充 `string_cast` 与 `numeric_cast` 的浮点往返转换测试，提升基础工具模块的可验证性。
```

## 不推荐写法

```text
[update] update code
[fix] 修点问题
chore: 修改一些东西
```

这些写法的问题在于：标签不标准、摘要不明确、正文缺少结构，后续不利于检索、审查和生成变更记录。

## 简要原则

1. 标题先说明“为什么提交”。
2. 正文再说明“具体改了什么”。
3. 优先保证提交粒度清晰，再追求描述完整。
4. 能拆分的提交尽量拆分，避免一个提交同时表达多个主题。
