# 单元测试命名与编写规范

本项目统一采用 **“场景 + 期望”** 的单测命名方式，并强制执行 **一个 case 只测一个目标**。

## 命名规则

`TEST_CASE` 名称使用中文，推荐句式：

- `在<场景>时，应该<期望结果>`
- `当<前置条件>时，应该<可观察行为>`

示例：

- `在传入相对路径时，应该返回对应的绝对路径`
- `在使用相同种子时，应该生成确定且一致的序列`

## 单目标规则（强制）

每个 `TEST_CASE` 只能验证 **一个测试目标**（一个行为或一个性质）。

### 合规示例

- 验证 `choice` 返回值来自源集合。
- 验证 `sample` 返回数量正确。

### 不合规示例

- 在同一个 case 里同时验证 `choice` 和 `sample`。
- 在同一个 case 里同时验证 `std` 哈希稳定性和 `FNV-1a` 哈希稳定性。

## 拆分原则

出现以下情况必须拆分 case：

1. 名称中出现“与/以及/并且”，且语义是不同目标。
2. 断言覆盖两个以上独立行为。
3. 一个行为失败不应影响另一个行为的定位。

## 标签规则

- 保留模块标签（如 `[coding]`、`[utility]`）。
- 重命名时只改 `TEST_CASE` 名称，不改既有标签语义。

## 评审检查清单

- [ ] 名称是否为“场景 + 期望”句式。
- [ ] 一个 `TEST_CASE` 是否仅对应一个目标。
- [ ] 是否需要拆分以提升失败定位能力。
- [ ] 是否保留并正确使用模块标签。
